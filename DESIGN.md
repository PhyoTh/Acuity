## 1. One Supabase Postgres for Both Auth + App Data

**Decision.** We decided to use a single Supabase project for both authentication and our PostgreSQL database, keeping all of our backend infrastructure centralized instead of splitting services apart.

When we were first planning the backend architecture using Claude, we learned several different appoaches we could take to set up the backend. Claude helped compare several different approaches, such as using a separate hosted Postgres instance versus keeping everything inside Supabase. One thing it highlighted was that the Supabase Python client is synchronous, meaning it could slow down or block our WebSocket-based real-time features. Since our platform relies heavily on live synchronization and interactive updates, we decided to build around an async stack using SQLAlchemy 2.0 async and asyncpg instead. For this decision, we would say that we and AI played an equal part in the final decision.

## 2. The Hallucination Injector 

**Decision.** This is one of the more creatives features where we allow intentional hallucination injections into the AI to ensure that candidates do not simply copy paste AI code and are knowledgable enough to catch AI errors. For this, we have two LLMs calls where the first call products the correct answer, while the second call subtle rewrites the answer to contain a flaw. 


This idea is ours (mine + Sithu's, from the original proposal).  We thought it was a creative, fun, and valuable addition to the project. The tradeoff is cost and latency since a hallucinated turn requires two LLM calls. Although we would prefer to have just one LLM call, we realized that AI does much better at making a correct thing slightly wrong rather than coming up with a slightly hallucinated output. Additionally, we wanted the interviewer to always spot the hallucination along with the correct answer which is easier to implement with a two step process. One thing that we do want to highlight is that we actually show the candidate when the AI has been injected with a hallucinator to be fully transparent with them.

## 3. One Token Budget for Interview Sessions

**Decision.** Each session has a single session-wide token budget that limits how much the candidate can use the AI. When the token runs out, the candidate can no longer communicate with the AI. 

Originally, our interview sessions had a maximum limit for how many times a candidate can ask the AI for help and a maximum token limit for each AI response. However, when testing the site, we found it confusing to understand what each quota did. We figured that it would be way simpler to just collaspe them down into a token budget since most people understand and care about token budget. We believe that this change made our user experience more intuitve for both the interviewer and the candidate. We believe that it was fully our decision to implement this design.

## 4. Overall UI

**Decision**. We treated AI-generated UI designs as starting points rather than final products and iterated on the interface to create a more polished and professional user experience. The final UI was inspired by company websites and HackerRank.

We are pretty proud of the work we put into the overall UI and user experience of the platform. Instead of simply accepting the first design generated by AI, we treated those outputs as starting points and went through many rounds of iterations to shape the website into something that matched what we wanted. Throughout the process, we carefully adjusted layouts, color schemes, and interaction patterns to create a cleaner and more polished experience. While we ultimately used Claude Design to help finalize portions of the interface, the final product reflects a significant amount of our own design thinking and decision-making. We drew inspiration from company websites as well as HackerRank to create a UI that feels both professional and modern. We would say we are very satisfied with the end result of the UI.
