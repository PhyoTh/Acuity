## Decision A — Where does app data live?

Supabase isn't just auth; it's a managed **Postgres + Auth** bundle. So the real question is "one database or two?"

| | **A1. Supabase for both** | **A2. Supabase Auth only + separate Postgres** |
|---|---|---|
| Infra to run | One DB, one connection string | Two systems (Supabase project + Render Postgres) |
| Auth ↔ data | Co-located; can FK to `auth.users`, use RLS | Reference users by UUID only across two DBs |
| Free tier | Generous; built-in table viewer/SQL editor (great for demoing telemetry) | Render free Postgres **expires after 90 days** → cost/migration |
| JWT verification | You still verify Supabase JWT in FastAPI | Same — you verify Supabase JWT in FastAPI |
| Control | Limited to Supabase's Postgres config/extensions | Full control over your Postgres |
| Lock-in | Coupled to Supabase (but it's plain Postgres → portable) | Cleaner boundary, more glue code |
| Watch out | Connection limits on free tier → use Supabase's transaction pooler | More credentials/dashboards to manage |

**My rec: A1 (Supabase for both).** For a class project shipping "auth + live URL," it's the least infra, has a generous free tier, gives you one migration target, and a built-in table browser to inspect telemetry live during demos. You keep *your own* tables (`interview_rooms`, `events`, `transcripts`, `scorecards`) in the same Postgres next to Supabase's `auth` schema — a clean, common pattern. The only real caveat (free-tier connection limits) is easily handled with one long-running FastAPI service + the pooler.

## Decision B — How does FastAPI talk to Postgres?

| | **B1. SQLAlchemy 2.0 async + Alembic** | **B2. Raw asyncpg + schema.sql** | **B3. Supabase Python client** |
|---|---|---|---|
| Fit for async WS server | Excellent (asyncpg driver, non-blocking) | Excellent (lowest-level async) | **Poor — supabase-py is synchronous**, would block the event loop |
| Migrations | Alembic: versioned, ordered, review-friendly | Hand-managed schema.sql, manual re-apply | Managed in Supabase UI/SQL |
| Two-dev schema changes | Safe (ordered migration files) | Risky (schema.sql merge conflicts) | OK but outside your repo |
| Type safety / boilerplate | Typed models, less query boilerplate | You map rows by hand | Fluent client, least code |
| Telemetry insert performance | Great | Best (raw) | Weaker, API round-trips |
| Cost | Heavier dep, SQLAlchemy 2.0 learning curve | Minimal deps | Couples data access to Supabase API |

**My rec: B1 (SQLAlchemy 2.0 async + Alembic).** The proposal explicitly wants *non-blocking async telemetry logging* on a WebSocket server — async SQLAlchemy + asyncpg nails that, Alembic gives you two-person schema coordination, and it's the production-grade default you asked for. **B3 is a real trap here**: supabase-py is sync and would stall the real-time loop. B2 works but loses migrations + type-safety.

**These compose cleanly:** B1 points SQLAlchemy/Alembic straight at the Supabase Postgres connection string (A1) and manages your tables there.
